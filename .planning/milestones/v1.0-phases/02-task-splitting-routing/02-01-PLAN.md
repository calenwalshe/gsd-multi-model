Add a new `<task_routing>` section to gsd-planner.md. Insert it AFTER the task generation instructions but BEFORE the revision mode section. This section defines the classification heuristic the planner must run on every task it generates.

The section must contain the following subsections in order:

**1. Task Routing Heuristic Overview**
- State: "After generating all tasks for a plan, run this heuristic on each task to assign `executor` and `confidence` attributes."
- State: "Every `<task>` element MUST have both `executor` and `confidence` attributes."
- State: "Missing executor defaults to claude in downstream consumers (Phase 3+)."

**2. Step 1: Pre-Check (checkpoint tasks)**
- Rule: If task `type` starts with `checkpoint:`, ALWAYS assign `executor="claude"` and `confidence="high"`.
- Skip all remaining steps for checkpoint tasks.

**3. Step 2: Type Shortcut (fast path)**
- Define keyword matching on task `<name>` AND `<action>` fields using compound verb+noun patterns:
  - Codex keywords (compound patterns): "write tests", "add tests", "create tests", "write test", "create script", "write script", "add config", "update config", "create config", "write docs", "update docs", "create documentation", "add fixture", "create stub", "create mock", "fix linting", "fix formatting", "generate CRUD", "write CRUD"
  - Claude keywords (single or compound): "architecture", "refactor", "redesign", "debug", "investigate", "optimize", "design", "integrate", "wire up", "coordinate", "multi-file refactor"
- If a Codex compound pattern matches: assign `executor="codex"`, `confidence="high"`, SKIP to Step 4.
- If a Claude keyword matches: assign `executor="claude"`, `confidence="high"`, SKIP to Step 4.
- If no match: proceed to Step 3.

**4. Step 3: 4-Signal Analysis (fallback for unmatched tasks)**
- SCOPE: Count files in `<files>` field.
  - 1 file = Codex-safe signal
  - 2-3 files = neutral (no signal)
  - 4+ files = Claude signal
- CLARITY: Analyze `<action>` field for deterministic success criteria.
  - Clear input/output, specific commands, deterministic behavior = Codex-safe signal
  - "Design", "decide", "evaluate", "investigate", "explore" = Claude signal
- ISOLATION: Check if task touches public APIs, shared types, cross-module contracts.
  - Internal-only changes = Codex-safe signal
  - Public API, shared interface, cross-service = Claude signal
- ERROR COST: Assess blast radius if task fails.
  - Caught by tests, low blast radius = Codex-safe signal
  - Blocks other work, production risk, hard to revert = Claude signal
- ROUTE based on signal count:
  - 4 Codex-safe signals = `executor="codex"`, `confidence="high"`
  - 3 Codex-safe signals = `executor="codex"`, `confidence="medium"`
  - 2 Codex-safe signals = `executor="claude"`, `confidence="medium"` (conservative default)
  - 1 or 0 Codex-safe signals = `executor="claude"`, `confidence="high"`
  - Any ambiguity present = `executor="claude"`, `confidence="low"`

**5. Step 4: Routing Summary**
- After ALL tasks are classified, present a routing summary table:
```
## Task Routing Summary

| Task | Executor | Confidence | Reason |
|------|----------|------------|--------|
| Task 1: ... | codex | high | Keyword: "write tests" |
| Task 2: ... | claude | high | 4+ files, architecture |

Overrides: Edit `executor` attribute in PLAN.md directly, or reply "override Task N to codex/claude".
```
- Include this table in the planner's `## PLANNING COMPLETE` return block so the orchestrator displays it.

**6. Revision Mode Override Preservation**
- Add to the existing revision mode instructions: "When revising an existing plan, read the current PLAN.md first. For tasks that already exist (matched by `<name>` field), NEVER change the `executor` or `confidence` attributes — they may have been manually overridden by the user. Only classify NEW tasks added during revision."

**Anti-patterns to document in the section:**
- NEVER block planning to ask about routing — decide and let user override afterward.
- NEVER re-classify existing tasks during revision runs.
- NEVER route checkpoint tasks to Codex.
- NEVER omit confidence — all tasks must have both executor AND confidence.

**Formatting requirements:**
- Use XML tags for the section: `<task_routing>` ... `</task_routing>`
- Keep the section under 120 lines (must fit within the 200-line guideline for agent files alongside other sections)
- Use clear numbered steps matching the structure above
