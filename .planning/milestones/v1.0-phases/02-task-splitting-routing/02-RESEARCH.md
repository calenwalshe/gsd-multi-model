## Summary

Phase 2 is a targeted extension to two existing artifacts: the `gsd-planner` agent prompt (`/home/agent/.claude/agents/gsd-planner.md`) and the PLAN.md XML schema (defined in `/home/agent/.claude/get-shit-done/templates/phase-prompt.md`). The classification logic does NOT require a new standalone module — it gets embedded in gsd-planner as a new section covering task-type shortcut matching, 4-signal heuristic evaluation, and routing summary presentation.

The current PLAN.md task element looks like `<task type="auto">`. The extension adds two new XML attributes: `executor="claude|codex"` and `confidence="high|medium|low"`. These attributes sit alongside the existing `type` attribute on the same element. The planner populates them during task generation. The execute-phase workflow reads them in Phase 3 to route tasks. The planner also runs a post-generation review step: present a routing summary table (task name | executor | confidence | reason), ask "any overrides?", and update executor attributes accordingly.

Override persistence is the key tricky area. When the planner runs in revision mode (re-invoked by plan-phase for checker iterations), it must not change executor attributes that were manually set. The safest approach is a "sticky executor" rule in the planner prompt: if a task already has `executor` set AND the value was not generated by the current planner run (detectable via confidence or a `user-override="true"` attribute), preserve it. The simpler user-facing path is direct PLAN.md XML editing — the planner never re-runs classification on existing plans unless the plan is being regenerated from scratch.

**Primary recommendation:** Implement Phase 2 as two file edits — add a `<task_routing>` section to `gsd-planner.md` and update the task element template in `phase-prompt.md`. One PLAN.md serves as the verification test case.

### Classification Logic Structure (to embed in gsd-planner.md)

```
TASK ROUTING HEURISTIC (run for each task after generating action content):

STEP 1: TYPE SHORTCUT (fast path — check task name and description)
  - Keywords → Codex: "test", "tests", "spec", "script", "config", "docs", "documentation", "fixture", "stub", "mock", "linting", "formatting", "CRUD", "generate"
  - Keywords → Claude: "architecture", "refactor", "redesign", "debug", "investigate", "optimize", "design", "integrate", "wire", "coordinate"
  - If keyword match found: assign executor + confidence="high", SKIP Step 2

STEP 2: 4-SIGNAL ANALYSIS (fallback for unlabeled tasks)
  - SCOPE: Count files in <files> field
    - 1 file → Codex-safe signal
    - 2-3 files → neutral
    - 4+ files → Claude signal
  - CLARITY: Does action section have deterministic success criteria?
    - Clear input/output, specific commands → Codex-safe signal
    - "Design", "decide", "evaluate", "investigate" → Claude signal
  - ISOLATION: Does task touch public APIs, shared types, cross-service contracts?
    - Internal only → Codex-safe signal
    - Public API / shared interface → Claude signal
  - ERROR COST: What happens if this fails?
    - Caught by tests, low blast radius → Codex-safe signal
    - Blocks other work, production risk → Claude signal

STEP 3: ROUTE
  - All 4 signals = Codex-safe → executor="codex", confidence="high"
  - 3 signals = Codex-safe → executor="codex", confidence="medium"
  - 2 signals = Codex-safe → executor="claude", confidence="medium" (conservative default)
  - 1 or 0 signals = Codex-safe → executor="claude", confidence="high"
  - Any Claude signal present + ambiguous → executor="claude", confidence="low"
```

### Routing Summary Presentation (post-plan-generation step)

After all tasks are tagged, present to user:

```
## Task Routing Summary

| Task | Executor | Confidence | Reason |
|------|----------|------------|--------|
| Task 1: Write unit tests | codex | high | Type shortcut: "tests" |
| Task 2: Design routing architecture | claude | high | 4 signals: multi-file, architecture keyword |
| Task 3: Update config | codex | medium | 3/4 signals Codex-safe |
| Task 4: Integrate with planner | claude | low | Ambiguous — defaulted to Claude |

Overrides: Edit PLAN.md executor attributes directly, or specify "override task N to codex/claude" and I'll update the file.
```

### Override Persistence Rule

When gsd-planner runs in revision mode (re-invoked to fix checker issues):
- If a task already exists in the PLAN.md with an executor attribute set, PRESERVE that attribute
- Only (re-)classify NEW tasks added during revision
- Never re-classify existing tasks — they may have been manually overridden

Implementation: In the planner's revision instructions, add: "If task already has `executor` attribute, do not change it."

### Anti-Patterns to Avoid

- **Blocking planning on routing uncertainty:** Never stop and ask user whether to route — decide and tag with low confidence. User can override afterward.
- **Re-classifying on revision:** Planner revision runs fix checker issues; they must not alter executor tags on existing tasks.
- **Omitting executor on checkpoint tasks:** Checkpoint tasks (human-verify, decision) should get `executor="claude"` since they require Claude interaction. Never route checkpoints to Codex.
- **Missing confidence attribute:** All tasks must have both `executor` AND `confidence`. Low-confidence tags are the user's signal to review.

## Common Pitfalls

### Pitfall 1: Checkpoint Tasks Routed to Codex

**What goes wrong:** `checkpoint:human-verify` or `checkpoint:decision` tasks get tagged `executor="codex"` because the heuristic sees "single file" and "clear criteria."
**Why it happens:** The heuristic only looks at scope/clarity signals, not task type.
**How to avoid:** Add a pre-check in Step 1: if `type` starts with `checkpoint:`, always assign `executor="claude"`, skip all other steps.
**Warning signs:** Any checkpoint task with executor="codex" in the generated plan.

### Pitfall 2: Revision Run Overwrites Manual Overrides

**What goes wrong:** User sets `executor="codex"` on a task manually. Plan-phase runs again (checker iteration), planner re-classifies and overwrites the override.
**Why it happens:** Planner prompt doesn't distinguish existing vs new tasks.
**How to avoid:** In revision mode prompt, add explicit rule: "Read existing PLAN.md first. For tasks that already exist (matched by task name), NEVER change the executor attribute."
**Warning signs:** User reports override not persisting across planner iterations.

### Pitfall 3: Schema Extension Breaks Existing Plans

**What goes wrong:** Adding `executor` and `confidence` as required attributes causes existing PLAN.md files (Phase 1 plans) to fail validation or cause executor errors when Phase 3 reads them.
**Why it happens:** Phase 3 execute routing will look for `executor` attribute; missing attribute = null dereference risk.
**How to avoid:** Phase 3 (not Phase 2) handles executor-absent tasks. Establish the convention now: missing `executor` defaults to `claude`. Phase 3 must handle this gracefully. Document in schema.
**Warning signs:** Phase 3 plan-checker flags Phase 1 PLAN.md files as missing required fields.

### Pitfall 4: Overly Broad Keyword Matching

**What goes wrong:** The word "test" appears in a task name like "Test the integration" — which is actually architecture work — and gets routed to Codex.
**Why it happens:** Keyword matching is too shallow; doesn't consider context.
**How to avoid:** Match compound patterns: "write tests", "add tests", "create tests" → Codex shortcut. Just "test" alone → fall through to 4-signal analysis. Use verb+noun patterns, not single words.
**Warning signs:** Claude-complexity tasks being tagged codex/high confidence in routing summary.

## Open Questions

1. **Does plan-phase.md orchestrator need a routing-summary presentation step?**
   - What we know: plan-phase spawns gsd-planner and then gsd-plan-checker; the routing summary could be presented either by gsd-planner (inside Task call) or by plan-phase orchestrator (after Task returns)
   - What's unclear: Whether gsd-planner's return output is surfaced directly to user, or filtered by orchestrator
   - Recommendation: Have gsd-planner include the routing summary in its structured return (`## PLANNING COMPLETE` block). Orchestrator displays it verbatim. No orchestrator changes needed.

2. **Should gsd-plan-checker validate executor attributes?**
   - What we know: Plan-checker verifies structural completeness. If executor is now expected, checker should flag missing attributes.
   - What's unclear: Whether updating the checker is in Phase 2 scope (CONTEXT.md doesn't mention it).
   - Recommendation: Add a brief note to the checker that executor+confidence are expected on non-checkpoint tasks. Minimal change — one bullet in checker's validation checklist.

3. **How does revision mode distinguish "existing task" vs "new task"?**
   - What we know: Revision prompt gives planner the existing PLAN.md plus checker issues. Planner reads both.
   - What's unclear: Whether task name is stable enough to use as identity key between revisions.
   - Recommendation: Task name IS the identity key for override persistence. Document this in the planner's revision instructions: "Match by task `<name>` field. Preserve executor on matched tasks."
