The skill must instruct Claude to execute the following steps IN ORDER:

**Step 1: Parse arguments**
- If `$ARGUMENTS` contains `--commits=N`, extract N as the diff scope (default: 5)
- Remaining arguments (after stripping --commits=N) become the focus area
- If no focus area provided, use "general code quality review"

**Step 2: Check Codex availability**
- Run `command -v codex` to check if Codex CLI is installed
- If NOT installed:
  - Print warning: "Codex CLI not installed. Skipping cross-model review."
  - Print install hint: "Install with: npm install -g codex-cli"
  - Print: "Proceeding with Claude-only review of any Codex-built code."
  - Set `CODEX_AVAILABLE=false`
- If installed: set `CODEX_AVAILABLE=true`

**Step 3: Read timeout configuration**
- Read `.planning/config.json` if it exists
- Extract `codex.timeout_seconds` (default: 300 = 5 minutes)
- This timeout applies to the `codex exec` subprocess

**Step 4: Gather review context**
- Read `.planning/STATE.md` for current phase and position
- Read `.planning/REQUIREMENTS.md` for expected behavior
- Run `git diff HEAD~{N} --stat` to get change summary (N from step 1)
- Run `git diff HEAD~{N}` to get full diff
- Read the latest phase SUMMARY and VERIFICATION files if they exist:
  - Glob for `.planning/phases/*-*/*-SUMMARY.md` and `*-VERIFICATION.md`
- Identify Codex-built changes: check git log for commits with "codex" in author or message

**Step 5: Build and run Codex review (if available)**
- Only if `CODEX_AVAILABLE=true`
- Construct the review prompt including:
  - Project requirements summary (from REQUIREMENTS.md)
  - Current phase context (from STATE.md)
  - Focus area (from arguments)
  - The diff output
  - Review checklist: bugs, security (OWASP top 10), missing test coverage, edge cases, convention violations per AGENTS.md, over-engineering
  - Output format: CRITICAL/WARNING/INFO findings
- Execute via Bash:
  ```bash
  timeout {timeout_seconds} codex exec --full-auto "{review_prompt}"
  ```
- Capture stdout and exit code
- If exit code is non-zero or timeout:
  - If timeout (exit 124): print "Codex review timed out after {N}m. Increase codex.timeout_seconds in .planning/config.json"
  - If other error: print "Codex review failed: {error}. Continuing with Claude-only review."
  - Set `CODEX_REVIEW_RESULT="INCOMPLETE"`
- If successful: parse the output for CRITICAL/WARNING/INFO findings

**Step 6: Claude reviews Codex-built code (bidirectional)**
- Check git log for Codex-authored commits (look for "codex" in author or "[codex]" in message)
- If Codex-built changes found:
  - Read those specific changes (git show or git diff for those commits)
  - Verify they meet specs in .planning/PLAN.md and REQUIREMENTS.md
  - Check for: incomplete implementations, missing edge cases, test gaps, architectural mismatches
  - Report Claude's findings with same CRITICAL/WARNING/INFO format
- If no Codex-built changes: note "No Codex-built changes found in last {N} commits"

**Step 7: Display combined results**
- Format as structured report:
  ```
  === CROSS-MODEL REVIEW RESULTS ===

  Scope: Last {N} commits | Focus: {focus_area}

  Codex reviewed Claude's work:  {PASS|FAIL|INCOMPLETE|SKIPPED}
    {list of findings with severity}

  Claude reviewed Codex's work:  {PASS|FAIL|N/A}
    {list of findings with severity}

  === RECOMMENDATION ===
  {one of: "Cross-review clean. Safe to advance." | "Minor issues found. Consider fixing." | "Critical issues found. Fix before advancing, then re-run /codex-review."}
  ```

**Error handling requirements (per locked decisions):**
- Codex CLI missing: graceful skip, NOT a failure
- Codex timeout: report, continue with Claude-only results
- Codex partial output: use whatever was captured, note incompleteness
- No git history (fewer than N commits): adjust to available commits with `git rev-list --count HEAD`
- Empty diff (no changes): report "No changes found in last {N} commits" and exit cleanly
- Missing .planning/ files: skip context gathering for that file, note in output

**Key formatting note:**
The SKILL.md IS the prompt. Write as executable instructions Claude follows step-by-step. Use numbered steps, bash code blocks, and explicit conditional logic.
