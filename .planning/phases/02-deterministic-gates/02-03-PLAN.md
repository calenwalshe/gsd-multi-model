**Part A: Create bin/gsd-tools-gate.cjs**

Create a standalone Node.js CLI script (not a module -- invoked directly) that wraps gate operations:

1. `node bin/gsd-tools-gate.cjs run [--plan-path <path>] [--raw]` -- Executes `bin/gate-check.sh` via child_process.execSync, captures stdout (JSON) and stderr (human-readable), returns the JSON result. Passes through `--plan-path` if provided.

2. `node bin/gsd-tools-gate.cjs check-architecture [--files <file1> <file2>...]` -- Executes `bin/validate-architecture.sh` directly. Useful for standalone architecture checks.

3. `node bin/gsd-tools-gate.cjs status` -- Reads `.planning/config.json` gates section and reports which gates are enabled/disabled.

Since gsd-tools.cjs lives in GSD base (`$HOME/.claude/get-shit-done/bin/`) and base updates would overwrite modifications, this is a standalone script in THIS project. The gate-check skill tells agents to invoke it directly.

**Part B: Create skills/gate-check/SKILL.md**

Create a skill (under 150 lines) that executor agents load during execute phase. It provides:

1. Description of the gate system (what it does, why it exists)
2. The modified task_commit protocol with gate check inserted between staging and committing:
   - After staging files, run `bash bin/gate-check.sh` (or `node bin/gsd-tools-gate.cjs run`)
   - If gates fail: print error, fix violations, re-stage, re-run gates
   - Only proceed to commit when gates pass
3. How to handle each gate type (lint, architecture, structural)
4. The `--plan-path` flag for structural tests
5. Emergency escape: document that setting `gates.enabled=false` in config.json skips all gates
